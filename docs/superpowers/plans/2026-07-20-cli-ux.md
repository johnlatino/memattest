# CLI UX Round Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Four usability fixes from watch-list manual testing: a `--project` convenience for external `adopt`/`unwatch`, a clearer derivation error, `install` watching only the shared settings file, and complete per-command help.

**Architecture:** All changes are in the CLI surface and the installer's onboarding step. A small shared helper resolves the memory directory from `--memory-dir`/`--project` for the external-file commands, reusing the installer's existing `derive_memory_dir`. No core, log-format, or keystore changes.

**Tech Stack:** Python ≥ 3.12, argparse, pytest. Spec: `docs/superpowers/specs/2026-07-20-cli-ux-design.md`.

## Global Constraints

- **venv only** - every `pytest` and `memattest` invocation uses `.venv\Scripts\...` (Windows dev machine). Branch: `cli-ux`.
- No new dependencies. No core/log-format/keystore changes; scheme stays `"v1"`.
- The hot `hook pre-tool-use` path imports nothing heavy: `derive_memory_dir` is lazy-imported inside the command handlers/helper, not at module top; `tests/test_cli.py::test_cli_module_import_stays_lightweight` must keep passing.
- `--memory-dir` and `--project` given together is a usage error. Precedence when only one is given: `--memory-dir` > `--project` > today's file-parent derivation (`adopt`) / required-flag error (`unwatch`).
- No walk-up or inference of the project - consistent with the existing strict rule in `_derive_memory_dir`.
- Wording, everywhere (docs, help strings, messages, commits): plain phrasing over fancy; **no em-dashes** (use single hyphens, commas, or restructure); "backend keystore" never bare "backend"; never "load-bearing"; never the informal self-testing term; no contrastive-reframe constructions; "procedure" not "ceremony".
- Commit messages: concise (short subject plus at most a one-or-two-sentence body); subject + body only, **no attribution lines**.
- **Guard phrase discipline:** the live PreToolUse guard denies any shell command containing the two-word phrases `memattest adopt`, `memattest install`, `memattest unwatch`, paths shaped like `.claude/settings*.json`, or the hook-disabling flag name. Run tests via pytest paths/keywords only. Test each command's `-h` through `cli.main(["adopt", "-h"])` (argparse raises `SystemExit(0)`), never by running the guarded phrase in a shell. Help/epilog example strings written into source via Edit may contain those phrases (the guard inspects executed commands, not source).

```python
def _memory_dir_from_flags(args) -> str | None:
    """Resolve --memory-dir / --project for adopt and unwatch.

    Returns the memory-dir string, or None when neither flag was given (the
    caller decides the fallback). Passing both is a usage error; a --project
    whose derived memory directory does not exist is an operational error.
    """
    project = getattr(args, "project", None)
    if project is not None and args.memory_dir is not None:
        raise MemAttestError("pass --memory-dir or --project, not both")
    if args.memory_dir is not None:
        return args.memory_dir
    if project is not None:
        from .integrations.claude_code.install import derive_memory_dir
        derived = derive_memory_dir(Path(project))
        if not derived.is_dir():
            raise MemAttestError(
                f"derived memory directory {derived} does not exist; pass "
                "--memory-dir explicitly, or run a Claude Code session in the "
                "project first so the directory exists"
            )
        return str(derived)
    return None
```
